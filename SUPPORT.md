# Support

Questions are welcome in this repository, but different types of questions should go through different doors.
Use the right doorway and the maintenance load turns from driftwood into a proper ship.

## 1. Which Channel to Use

### Bugs and Regressions

Please use GitHub Issues and prefer the Bug template.
This covers cases such as "it used to work and now it doesn't", "the documented behavior does not match reality", or "a configuration or workflow fails reproducibly".

### New Features and Product Suggestions

Please use GitHub Issues and prefer the Feature Request template.
This covers new capabilities, interface changes, deployment support, ecosystem integrations, or stronger governance features.

### Usage Questions and Help Requests

If GitHub Discussions is enabled, prefer Discussions.
If Discussions is not enabled, use the Question template.

### Security Vulnerabilities

**Do not open a public issue.**
Please read and follow `SECURITY.md`.

### Trademark or Brand Use

If your question concerns the project name, logo, derivative distribution naming, marketing presentation, or external endorsement, please read `TRADEMARKS.md` first.

## 2. What to Include When Asking for Help

- The branch, tag, or commit you are using
- Deployment method, such as Docker Compose, Kubernetes, or a local development environment
- Operating system, container runtime, model provider, and messaging channel
- Relevant configuration snippets, with secrets redacted
- Minimal reproduction steps
- Expected result and actual result
- Key logs, screenshots, or stack traces, with sensitive data removed

## 3. Please Do Not Submit These Things

- Access tokens, API keys, cookies, passwords, or private keys
- Internal enterprise chat records, customer data, or production database fragments
- Logs containing real personal data or business secrets
- Unredacted audit evidence

## 4. Support Scope

Right now we mainly focus on:

- Problems on the default branch or recent versions
- Issues directly related to the project's core direction
- Core areas such as containerized deployment, multi-user isolation, memory systems, Skill sharing, audit governance, and enterprise messaging integrations

We generally do not commit to supporting:

- Private forks that have not been kept in sync for a long time
- Heavily rewritten downstream versions
- Third-party infrastructure problems that are not directly part of this project
- Reports with no reproducible information, logs, or context

## 5. Response Expectations

This project does not currently offer a commercial SLA.
Support is provided on a best-effort basis and depends on maintainer availability.

If you want your issue handled faster, the most helpful message is rarely "any update?". It is a complete reproduction record.
